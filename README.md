# awgram

🇷🇺 Русский · [🇬🇧 English](README.en.md)

Telegram-бот на Rust для управления клиентами [AmneziaWG](https://amnezia.org/) прямо
с телефона: добавить/удалить клиента, посмотреть список и трафик — без SSH.

## Что это и как работает

`awgram` — один бинарник (`tokio` + `teloxide 0.17`, long polling, без webhook),
который живёт на том же VPS, что и сам VPN. Бот не трогает конфигурацию
AmneziaWG напрямую — он вызывает штатный скрипт `manage_amneziawg.sh` (тот же,
которым вы управляете VPN руками) с подкомандами `add` / `remove` / `list` /
`stats` и флагом `--json`, и рендерит результат в виде inline-меню в Telegram.

Архитектурно это два независимых слоя:

- `vpn/` — вызывает скрипт через `tokio::process::Command` (без shell, без
  строкового форматирования команды) и разбирает его JSON-вывод в типы из
  `vpn/model.rs`. Ничего не знает про Telegram.
- `bot/` — строит меню, обрабатывает апдейты и рендерит ответы. Ничего не
  знает про детали скрипта.

Доступ к боту ограничен списком `admin_ids` из конфига: любой апдейт от
пользователя не из этого списка получает «⛔ Доступ запрещён» и не выполняет
ни одной операции над VPN. Токен бота и содержимое `.conf`/QR-кодов клиентов
никогда не попадают в логи.

## Сборка

Нужен стабильный Rust (edition 2021) и `cargo`.

```bash
cargo build --release
```

Бинарник появится в `target/release/awgram`.

TLS реализован на **rustls** (без OpenSSL), поэтому у бинарника нет системной
зависимости от `libssl`.

### Статические Linux-бинарники (для деплоя на VPS)

Готовый портативный бинарник собирается одной командой (нужен Docker):

```bash
./scripts/build-musl.sh          # amd64 (по умолчанию)
./scripts/build-musl.sh arm64    # aarch64
./scripts/build-musl.sh all      # обе архитектуры
```

Скрипт собирает в Docker-контейнере полностью статический ET_EXEC
(`x86_64-` / `aarch64-unknown-linux-musl`; `crt-static` +
`relocation-model=static` заданы в `.cargo/config.toml`, rustls) и кладёт его
в `dist/awgram-linux-amd64` / `dist/awgram-linux-arm64`. Такой бинарник не
зависит ни от glibc, ни от `libssl`, ни от musl-загрузчика — запускается на
любом Linux своей архитектуры (Ubuntu/Debian/Alpine). Проверка:
`ldd dist/awgram-linux-amd64` → `not a dynamic executable`.

GitHub Release (по тегу `v*`) собирает оба бинарника через
[cross](https://github.com/cross-rs/cross) и прикладывает их вместе с
`sha256`-контрольными суммами.

Перенос на сервер:

```bash
scp dist/awgram-linux-amd64 root@SERVER:/usr/local/bin/awgram
ssh root@SERVER chmod +x /usr/local/bin/awgram
```

> Сборка «чужой» для хоста архитектуры идёт под эмуляцией qemu (на Apple
> Silicon: arm64 — нативно, amd64 — эмуляция) — медленнее, но результат
> корректный. Кэш cargo хранится в docker-volume
> `awgram-cargo-registry-<arch>`, поэтому повторные сборки быстрее.

## Установка

Предполагается, что `manage_amneziawg.sh` уже установлен и рабочий (см.
документацию самого AmneziaWG-инсталлятора), обычно по пути
`/root/awg/manage_amneziawg.sh`.

### Быстрая установка (рекомендуется)

```bash
curl -fsSL https://raw.githubusercontent.com/ekuraev/awgram/main/install.sh | bash
```

Интерактивный установщик спросит язык (RU/EN), режим (root или hardened),
токен и ID администраторов, скачает бинарник под вашу архитектуру
(amd64/arm64) из последнего релиза с проверкой sha256 и запустит сервис.
Полностью автоматическая установка — флагами:

```bash
curl -fsSL https://raw.githubusercontent.com/ekuraev/awgram/main/install.sh \
  | bash -s -- install --lang ru --mode root --token 'ТОКЕН' --admins 111111111 --yes
```

После установки доступна команда `awgram-setup`:
`update` (обновление до последнего релиза), `config` (токен/админы/пути),
`status`, `uninstall`, `help`.

### Ручная установка

1. Скопируйте бинарник:

   ```bash
   sudo cp target/release/awgram /usr/local/bin/awgram
   sudo chmod 755 /usr/local/bin/awgram
   ```

2. Создайте директорию конфига и скопируйте пример:

   ```bash
   sudo mkdir -p /etc/awgram
   sudo cp deploy/config.example.toml /etc/awgram/config.toml
   sudo chmod 640 /etc/awgram/config.toml
   ```

   Отредактируйте `/etc/awgram/config.toml`: заполните `admin_ids` (см.
   раздел «Настройка Telegram» ниже). Поле `bot_token` в этом файле можно
   оставить пустым — токен лучше хранить отдельно (следующий шаг).

3. Создайте файл окружения с токеном и выставьте права `600`, чтобы его не
   мог прочитать никто, кроме владельца:

   ```bash
   sudo bash -c 'echo "AWGRAM_TOKEN=<ваш_токен_от_BotFather>" > /etc/awgram/env'
   sudo chmod 600 /etc/awgram/env
   ```

   Юнит `awgram.service` подхватывает этот файл через `EnvironmentFile=-/etc/awgram/env`
   (ведущий `-` означает «не падать, если файла нет» — но без токена бот всё
   равно откажется стартовать, см. ниже).

4. Установите systemd-юнит:

   ```bash
   sudo cp deploy/awgram.service /etc/systemd/system/awgram.service
   sudo systemctl daemon-reload
   ```

Токен можно указать либо через `AWGRAM_TOKEN` в `/etc/awgram/env` (приоритет
выше), либо прямо в `config.toml` как `bot_token = "..."`. Если ни там, ни там
токена нет, бот при старте выведет ошибку в лог и завершится с кодом 1.

## Настройка Telegram

1. Откройте в Telegram [@BotFather](https://t.me/BotFather), выполните
   `/newbot`, следуйте инструкциям — получите токен вида
   `123456789:AAExampleTokenValue`.
2. Узнайте свой числовой Telegram user ID через
   [@userinfobot](https://t.me/userinfobot) — просто напишите ему `/start`,
   он ответит вашим ID.
3. Впишите этот ID в `admin_ids` в `config.toml` (можно перечислить
   несколько администраторов через запятую: `admin_ids = [111111111, 222222222]`).
   Только пользователи из этого списка смогут пользоваться ботом; все прочие
   апдейты отклоняются с сообщением «⛔ Доступ запрещён» и записью `warn` в
   лог.

## Режимы привилегий

Бот выполняет `manage_amneziawg.sh`, которому обычно нужны root-права для
работы с интерфейсом WireGuard/AmneziaWG и файлами в `/root/awg`. Есть два
поддерживаемых способа это организовать.

### Простой режим (сервис от root)

- В `awgram.service` оставьте юнит как есть (без `User=`) — процесс
  выполняется от `root`, как и написано в комментарии юнита.
- В `config.toml`: `sudo_prefix = ""`.
- Плюс: минимум настройки. Минус: компрометация бота (например, через баг в
  парсинге Telegram-апдейтов) даёт полный root на сервере, где крутится VPN.

### Hardened-режим (отдельный пользователь + sudoers)

Более безопасный вариант — процесс бота работает от непривилегированного
пользователя, и только вызов конкретного скрипта разрешён через `sudo` без
пароля.

1. Создайте системного пользователя без домашней директории и шелла:

   ```bash
   sudo useradd --system --no-create-home --shell /usr/sbin/nologin awgram
   ```

2. Разрешите этому пользователю запускать **только** `manage_amneziawg.sh` от
   root без ввода пароля. Создайте файл `/etc/sudoers.d/awgram`
   (через `sudo visudo -f /etc/sudoers.d/awgram`) со строго одной строкой:

   ```
   awgram ALL=(root) NOPASSWD: /root/awg/manage_amneziawg.sh
   ```

   Проверьте синтаксис (`visudo` делает это автоматически при сохранении) и
   права на файл (`0440`).

3. В `config.toml` выставьте `sudo_prefix = "sudo"` — бот тогда будет
   запускать `sudo /root/awg/manage_amneziawg.sh <subcommand> ...` вместо
   прямого вызова.

4. В `/etc/systemd/system/awgram.service` добавьте строку `User=awgram` в
   секцию `[Service]` (после `Type=simple`).

5. **Важно — доступ на чтение файлов.** Скрипт кладёт `.conf` и QR-коды
   клиентов в `clients_dir` (по умолчанию `/root/awg`), а бот читает их
   напрямую (без sudo) с диска, чтобы отправить в Telegram через
   `send_document`/`send_photo`. Каталог `/root/awg` пользователю `awgram`
   недоступен по определению (он внутри `/root`). Варианты решения:
   - перенести/дублировать вывод скрипта в директорию, читаемую пользователем
     `awgram` (изменить `clients_dir` в `manage_amneziawg.sh` и в
     `config.toml` на что-то вроде `/var/lib/awgram/clients` с правами,
     позволяющими группе/пользователю `awgram` читать файлы);
   - либо выдать пользователю `awgram` точечный ACL на чтение этой
     директории (`setfacl -R -m u:awgram:rx /root/awg`).

     Без одной из этих мер шаги «Добавить клиента» и «Конфиг» будут падать с
     ошибкой чтения файла, даже если сам вызов скрипта через sudo прошёл
     успешно.

6. Примените изменения:

   ```bash
   sudo systemctl daemon-reload
   ```

## Запуск

```bash
sudo systemctl enable --now awgram
```

Проверить статус и посмотреть логи в реальном времени:

```bash
sudo systemctl status awgram
sudo journalctl -u awgram -f
```

Уровень логирования регулируется стандартной переменной `RUST_LOG`
(`tracing-subscriber` с `EnvFilter`, по умолчанию `info`) — её можно добавить
в `/etc/awgram/env`, например `RUST_LOG=debug`.

## Язык / Language

Бот двуязычный (русский/английский), выбор — per-admin (у каждого
администратора из `admin_ids` свой язык, независимо от остальных).

- **Первый запуск.** При первом `/start` от администратора, для которого язык
  ещё не сохранён, бот показывает экран выбора языка
  («🌐 Выберите язык / Choose language:») с двумя кнопками — 🇷🇺 Русский и
  🇬🇧 English. После выбора всё меню и последующие сообщения рендерятся на
  выбранном языке.
- **Смена языка.** ⚙️ Настройки → в верхнем ряду кнопки 🇷🇺 Русский /
  🇬🇧 English — нажатие сразу переключает язык интерфейса и обновляет экран
  настроек в новом языке.
- Язык хранится по числовому Telegram user ID администратора (`langs` в
  `state_file`, см. ниже), поэтому у разных админов на одном боте может быть
  разный язык одновременно.
- Если язык ещё не выбран (карта `langs` не содержит запись для этого `uid`),
  внутренние тексты по умолчанию рендерятся на русском (`Lang::Ru` — дефолт),
  но в чат в этом случае в первую очередь показывается сам экран выбора языка.

## PresharedKey (PSK)

AmneziaWG/WireGuard поддерживает дополнительный симметричный ключ
(PresharedKey) поверх обычной пары ключей — повышает устойчивость к будущим
квантовым атакам на Diffie-Hellman. Бот умеет включать его при создании
клиента через флаг `--psk`, который передаётся `manage_amneziawg.sh`.

- **Глобальный дефолт.** ⚙️ Настройки → кнопка «PSK: вкл ✅ / PSK: выкл ⬜»
  переключает значение по умолчанию, которое будет предложено при следующем
  добавлении клиента. Значение хранится в `state_file` (поле `psk_default`) и
  общее для всех администраторов бота (в отличие от языка).
- **Переопределение при добавлении.** В диалоге ➕ Добавить, после выбора
  срока действия, бот показывает шаг «PresharedKey (по умолчанию: вкл/выкл).
  Создать клиента:» с двумя кнопками — «🔐 С PSK» и «🔓 Без PSK». Кнопка,
  соответствующая текущему глобальному дефолту, идёт первой. Выбор на этом
  шаге действует только для создаваемого клиента и не меняет глобальную
  настройку.
- Если выбрано «С PSK», бот вызывает
  `manage_amneziawg.sh add <имя> [--expires=<срок>] --psk`; в этом случае в
  сгенерированном `.conf`-файле клиента в секции `[Interface]`/`[Peer]`
  появляется строка `PresharedKey = ...` (проверяется открытием файла —
  секрет в чат отдельно не выводится).

## Backup / Restore

Раздел 💾 Бэкап в главном меню управляет резервными копиями состояния
AmneziaWG, которые создаёт и хранит сам `manage_amneziawg.sh` (подкоманды
`backup`/`restore`) в каталоге `clients_dir/backups/` на сервере — бот только
вызывает скрипт и читает список файлов оттуда.

- **Создать бэкап.** 💾 Бэкап → «➕ Создать бэкап» — бот запускает
  `manage_amneziawg.sh backup`, затем берёт самый свежий по времени
  изменения `*.tar.gz` из `clients_dir/backups/` и присылает подтверждение
  с именем файла (`✅ Бэкап создан: <имя>.tar.gz`) вместе с карточкой этого
  бэкапа (кнопки «📥 Скачать» / «♻️ Восстановить»).
- **Список бэкапов.** 💾 Бэкап → «📃 Список бэкапов» — список архивов из
  `clients_dir/backups/`, отсортированный по времени изменения (сначала
  новые); если каталог пуст — «Бэкапов пока нет.». Каждая строка — кнопка с
  именем архива, ведущая на карточку конкретного бэкапа по его индексу в
  этом списке (0 = самый новый).
- **Скачать.** На карточке бэкапа кнопка «📥 Скачать» отправляет файл
  `.tar.gz` документом прямо в чат (`send_document`), без парсинга
  содержимого — можно сохранить локально как резервную копию вне сервера.
- **Восстановить.** Кнопка «♻️ Восстановить» на карточке бэкапа запрашивает
  подтверждение («♻️ Восстановить из `<имя>.tar.gz`? Текущее состояние будет
  заменено.») с кнопками «✅ Да» / «⬅️ В меню» — операция необратима
  (перезаписывает текущее состояние клиентов AmneziaWG), поэтому без
  подтверждения не выполняется. После подтверждения бот вызывает
  `manage_amneziawg.sh restore <путь-к-архиву>` и присылает «✅ Восстановление
  завершено.». Выбор бэкапа для восстановления всегда идёт по индексу в
  актуальном `list_backups()`, а не по произвольному пути — это исключает
  path traversal через имя файла.

## Проверка / Check

🩺 Проверка в главном меню запускает встроенную самодиагностику сервера —
подкоманду `manage_amneziawg.sh check`.

- В отличие от остальных операций, `check` может завершиться с ненулевым
  кодом выхода (обычно код `1`), если скрипт обнаружил проблемы
  (например, не запущен интерфейс, не совпадает конфигурация) — это **не**
  трактуется ботом как ошибка выполнения: и stdout, и код выхода
  анализируются, но сообщение об ошибке пользователю не показывается, пока
  сам вывод не пуст. Бот отправляет весь stdout `check` в чат как есть.
- Вывод оборачивается в `<pre>` (HTML `parse_mode`), поэтому сохраняет
  моноширинное форматирование скрипта и не пытается парситься как разметка;
  спецсимволы (`<`, `>`, `&`) в выводе экранируются перед отправкой, чтобы
  случайные `<...>` в диагностике не ломали HTML-сообщение.
- Пока идёт проверка, бот показывает промежуточное сообщение
  «⏳ Проверяю сервер…».

## Персистентность настроек (state_file)

Язык каждого администратора и глобальный дефолт PSK хранятся не в оперативной
памяти, а в JSON-файле, путь к которому задаётся полем `state_file` в
`config.toml` (по умолчанию `/etc/awgram/state.json`, см.
`deploy/config.example.toml`). Формат — простой JSON вида
`{"psk_default": true, "langs": {"111111111": "ru"}}`. Запись атомарна
(сначала во временный `state.json.tmp`, затем `rename` поверх основного
файла), поэтому обрыв процесса в середине записи не портит файл. При старте
бот читает `state_file`, если он существует и парсится — иначе стартует с
пустым состоянием (язык не выбран ни у кого, PSK-дефолт выключен) и создаёт
файл при первом изменении настроек. Поэтому смена языка/PSK-дефолта переживает
рестарт бота и сервера — ничего дополнительно настраивать не нужно, но
директория, где лежит `state_file` (по умолчанию `/etc/awgram`), должна
существовать и быть доступна на запись процессу бота (см. режимы привилегий
выше, если бот работает не от root).

## HTML-разметка сообщений

Все сообщения бота используют Telegram `parse_mode = HTML` (`ParseMode::Html`
в `teloxide`), а не Markdown/MarkdownV2. Любые динамические значения, которые
попадают в текст сообщения (имя клиента, URI для импорта, вывод `check`,
имена файлов бэкапов), предварительно экранируются функцией
`i18n::html_escape` (`&` → `&amp;`, `<` → `&lt;`, `>` → `&gt;`, в таком
порядке — иначе получилось бы двойное экранирование `&`). Разметка вроде
`<b>жирный</b>` или `<code>моноширинный</code>` в самих шаблонах сообщений
при этом работает как обычно. Секреты (токен бота, приватные ключи из
`.conf`, PresharedKey, stderr скрипта) в текст сообщений никогда не
подставляются напрямую — `.conf`/QR отправляются файлом, а ошибки скрипта
превращаются в общие локализованные фразы через `i18n::error_text`.

Пример карточки клиента (`👥 Клиенты` → клиент), как она реально
рендерится на русском при `parse_mode = HTML`:

```
👤 <b>alice</b>
Статус: Активен
IP: 10.8.0.2
Трафик:  ↓ 1.2 MB   ↑ 340 KB
Рукопожатие: 2 мин назад
Действует: 12д 4ч
```

Клавиатура под карточкой: «📄 Конфиг» / «🗑 Удалить», затем «⬅️ В меню». На
английском те же поля выглядят как `Status:`, `Traffic:`, `Handshake:`,
`Expires:`.

## Smoke-чеклист

Это ручной чек-лист для первого реального прогона на тестовом (или боевом)
сервере с уже настроенным AmneziaWG. Отмечайте пункты по ходу — здесь
результаты **не** проставлены, это инструкция для оператора, а не отчёт о
прогоне.

1. **Установка и старт.** Установить бота по шагам выше, запустить сервис.
   `journalctl -u awgram -f` должен показать сообщения `конфиг загружен` и
   `запуск long polling`, без ошибок.
2. **Авторизация.** `/start` от администратора (ID из `admin_ids`) → бот
   показывает главное меню (`👥 Клиенты`, `➕ Добавить`, `📊 Статистика`).
   `/start` от постороннего пользователя → бот отвечает «⛔ Доступ запрещён»,
   в логе появляется `warn`-запись; ни одна операция над VPN не выполняется.
3. **Добавление клиента.** `➕ Добавить` → ввести имя клиента → выбрать срок →
   бот присылает `.conf`-файл, QR-код (если он сгенерирован скриптом) и
   текстовое сообщение со ссылкой-URI для импорта. `add` не печатает JSON —
   бот запускает `manage_amneziawg.sh add <имя> [--expires=<срок>]` и затем
   читает созданные файлы из `clients_dir`. Если бот не находит `.conf` —
   проверьте, что `clients_dir` в конфиге указывает на каталог, куда скрипт
   кладёт файлы (по умолчанию `/root/awg`).

   **Защита от дубликатов.** Если клиент с введённым именем уже существует,
   бот не перезаписывает его молча — вместо этого показывается предупреждение
   с кнопками «♻️ Пересоздать» / «⬅️ В меню». При выборе «Пересоздать» бот
   спрашивает срок и PSK заново, затем удаляет старого клиента и создаёт
   нового (новые ключи, новый IP). Существование проверяется через
   `list --json` — если этот вызов падает, создание не блокируется
   (fail-open, с `warn` в логе).

   **Детект «тихого пропуска» скрипта.** Upstream `manage add` при
   существующем имени пишет warning, пропускает клиента и завершается
   с кодом 0 — для бота это выглядело бы как успех, и он отправил бы
   старый `.conf` как якобы новый (без запрошенных срока и PSK). Бот
   сравнивает отпечаток `<имя>.conf` (mtime, размер, inode) до и после
   `add`: если файл существовал и не изменился — создание было пропущено.
   В этом случае (гонка между проверкой и `add`, либо fail-open выше)
   бот показывает то же предупреждение с кнопкой «♻️ Пересоздать» вместо
   отправки устаревшего конфига.
4. **Список клиентов.** `👥 Клиенты` → список (`list --json`: `name`, `ip`,
   `client_ipv6`, `status`, `status_code`). Карточка клиента строится из
   `stats --json` (`rx`, `tx`, `last_handshake`, `status`) и показывает статус
   и трафик (↓/↑). Кнопка «Конфиг» повторно присылает `.conf`, QR и URI для
   уже существующего клиента.

   **Метка срока в списке.** Временные клиенты помечаются в списке меткой
   «⏳ 6д» (дни/часы до истечения); истёкший, но ещё не удалённый cron'ом —
   «⏳ истёк». Бессрочные клиенты показываются без метки. Срок читается из
   `clients_dir/expiry/<имя>` (тот же источник, что и карточка клиента).

   **Перевыпуск конфигов (regen).** Кнопка «🔄 Перевыпустить» в карточке
   клиента перегенерирует `.conf`/QR/URI (ключи и IP сохраняются) и присылает
   свежие файлы. Кнопка «🔄 Перевыпустить всех» внизу списка клиентов — после
   подтверждения запускает `regen` для всех; вариант «🔀 + сброс маршрутов»
   добавляет `--reset-routes` (замена индивидуальных AllowedIPs глобальным
   режимом сервера — нужно после смены режима маршрутизации). Массовый
   перевыпуск выполняется с утроенным таймаутом; при частичных ошибках бот
   сообщает «завершено с ошибками», детали — в логах сервера.
5. **Статистика.** `📊 Статистика` → сводка: всего клиентов, активных,
   суммарный трафик.
6. **Удаление.** Удаление клиента → запрос подтверждения → после
   подтверждения клиент пропадает из списка `👥 Клиенты`.
7. **Валидация произвольного срока.** В поле «✏️ Свой» (кастомный срок)
   ввод вида `10d` принимается и используется; мусорный ввод (пустая строка,
   `10`, `10x`, `1.5d`, `10d;ls` и т.п.) отклоняется с понятным сообщением об
   ошибке, без обращения к скрипту.

### Замечания к первому реальному прогону

- **(a) Схема `--json` — сверена с исходником скрипта. ✅** Структуры
  `Client`/`AddResult` в `src/vpn/model.rs` приведены в соответствие с
  реальным `manage_amneziawg.sh`: `list --json` (`name`, `ip`, `client_ipv6`,
  `status`, `status_code`), `stats --json` (те же плюс `rx`, `tx`,
  `last_handshake`), а `add` JSON не печатает — результат собирается из
  файлов. Если в будущем формат вывода скрипта изменится, обновите структуры
  и фикстуры в `src/vpn/model.rs` и прогоните `cargo test`.
- **(b) HTML вместо MarkdownV2 для URI. ✅ Решено.** `send_client_files` в
  `src/bot/render.rs` теперь отправляет ссылку-URI клиента с
  `parse_mode = ParseMode::Html` (через `i18n::import_link`, оборачивающую
  URI в `<code>...</code>` после `html_escape`), а не MarkdownV2 — см. раздел
  «HTML-разметка сообщений» выше. Это касается вообще всех сообщений бота,
  не только URI: MarkdownV2 в проекте больше не используется.

## Smoke-чеклист (v2)

Ручной чек-лист для проверки фич второй итерации (язык, PSK, бэкап/восстановление,
диагностика, персистентность) на тестовом (или боевом) сервере. Как и раздел
выше, это инструкция для оператора — пункты **не выполнялись** в рамках этой
задачи (нет доступа к живому серверу с AmneziaWG из среды, где готовились эти
изменения), результаты здесь не проставлены.

1. **Выбор языка при первом `/start`.** Убрать/переименовать существующий
   `state_file`, чтобы админ считался «новым» → `/start` → должен появиться
   экран «🌐 Выберите язык / Choose language:» с кнопками 🇷🇺 Русский /
   🇬🇧 English. Выбрать один → главное меню должно отрисоваться на выбранном
   языке (проверить подписи кнопок «👥 Клиенты»/«👥 Clients» и т.д.).
2. **Смена языка в настройках.** ⚙️ Настройки → нажать кнопку другого языка
   → текст экрана настроек и клавиатура должны переключиться немедленно, без
   повторного `/start`. Открыть главное меню — оно тоже должно быть на новом
   языке.
3. **PSK-дефолт в настройках + переопределение в add.**
   - ⚙️ Настройки → переключить «PSK: вкл ✅ / PSK: выкл ⬜» → значение должно
     сохраниться (см. пункт 7 ниже про `state.json`).
   - ➕ Добавить → имя → срок → на шаге PSK кнопка, соответствующая текущему
     глобальному дефолту, должна идти первой; создать клиента с явным «🔐 С
     PSK» (даже если дефолт «выкл») → открыть присланный `.conf`-файл и
     убедиться, что в нём присутствует строка `PresharedKey = ...`. Повторить
     с «🔓 Без PSK» → убедиться, что `PresharedKey` в конфиге отсутствует.
4. **Карточка клиента (HTML, локализация).** `👥 Клиенты` → выбрать клиента →
   карточка должна отрендериться жирным именем (`<b>...</b>` → жирный текст,
   не сырые теги), с полями «Статус/Status», «Трафик/Traffic»,
   «Рукопожатие/Handshake», «Действует/Expires» на текущем языке; кнопки
   «📄 Конфиг»/«🗑 Удалить» — тоже на этом языке.
5. **Backup: создание, скачивание, список, восстановление.**
   - 💾 Бэкап → «➕ Создать бэкап» → должно прийти подтверждение с именем
     `.tar.gz`-архива и карточка с кнопками «📥 Скачать»/«♻️ Восстановить».
   - На карточке нажать «📥 Скачать» → в чат должен прийти файл `.tar.gz`
     документом (проверить, что он открывается/распаковывается).
   - 💾 Бэкап → «📃 Список бэкапов» → должны быть перечислены все созданные
     архивы (новые сверху).
   - Открыть архив из списка → «♻️ Восстановить» → должен появиться запрос
     подтверждения с именем архива → подтвердить «✅ Да» → должно прийти
     «✅ Восстановление завершено.»; после этого `👥 Клиенты` должен
     отражать состояние на момент бэкапа.
6. **Проверка (check).** 🩺 Проверка → должно прийти сообщение
   «⏳ Проверяю сервер…», а затем результат в моноширинном блоке (`<pre>`) —
   проверить, что блок не ломает разметку сообщения, даже если в выводе
   скрипта встречаются символы `<`/`>`/`&`. Если `manage_amneziawg.sh check`
   на этом сервере обнаруживает проблему (выходит с кодом 1) — сообщение
   всё равно должно прийти с текстом проблемы, а не показать общую ошибку
   выполнения.

   **Диагностика.** Кнопка «🔬 Диагностика» в главном меню запускает
   `manage diagnose` (публичный IP, состояние модуля, известные операторские
   проблемы) и показывает отчёт; длинный вывод обрезается до лимита Telegram.
7. **Персистентность после рестарта.** Записать текущий язык и состояние
   PSK-дефолта → `cat /etc/awgram/state.json` (или путь из `state_file` в
   конфиге) и убедиться, что JSON содержит `"psk_default"` в ожидаемом
   значении и запись в `"langs"` с ID администратора и выбранным языком →
   `sudo systemctl restart awgram` → `/start` от того же администратора →
   меню должно сразу открыться на прежнем языке (без повторного запроса
   выбора языка), а следующий заход в ⚙️ Настройки — показать прежнее
   значение PSK-дефолта.
