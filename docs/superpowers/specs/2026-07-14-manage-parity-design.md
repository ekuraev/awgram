# Паритет с manage-скриптом: regen, diagnose, метка срока в списке

Дата: 2026-07-14
Статус: утверждено (брейнсторм с владельцем проекта)

## Цель

Открыть в боте возможности `manage_amneziawg.sh`, которые уже есть в скрипте,
но недоступны из Telegram: перевыпуск клиентских конфигов (`regen`, включая
массовый с `--reset-routes` из upstream v5.19.1), расширенная диагностика
(`diagnose`) и видимость срока действия временных клиентов в списке.

## Скоуп

Входит:

1. **regen одного клиента** — кнопка «🔄 Перевыпустить» в карточке клиента;
   после перевыпуска бот присылает свежие `.conf`, QR и URI.
2. **Массовый regen** — кнопка «🔄 Перевыпустить всех» внизу списка клиентов,
   с экраном подтверждения и опцией «+ сброс маршрутов» (`--reset-routes`).
3. **diagnose** — кнопка «🔬 Диагностика» в главном меню рядом с «🩺 Проверка»,
   запуск `manage diagnose` без параметров, вывод в `<pre>`.
4. **Метка срока в списке клиентов** — «⏳ 6д» / «⏳ истёк» рядом с именем
   временного клиента в кнопках списка (карточка уже показывает срок —
   реализовано ранее через `format_expiry`).

Не входит (решения брейнсторма):

- **Batch add** — отклонено (сценарий редкий, добавление по одному достаточно).
- **Продление срока существующего клиента** — отложено: в `manage` нет команды
  изменения expiry (срок задаётся только при `add --expires`); писать
  expiry-файл в обход скрипта не хотим. Возможный путь — PR в upstream
  (`manage expire <name> <duration>`), отдельной задачей.
- **Выбор оператора для diagnose** (`--carrier=NAME`) — отклонено, запуск без
  параметров покрывает основной сценарий.
- Фоновый мониторинг и алерты — второй подпроект, отдельная спека после этого.

## Архитектура

Подход: три независимых вертикальных среза по существующим паттернам кодовой
базы (метод в `Vpn` → строки в `i18n` → кнопки в `menu` → ветка в `handlers`
→ тесты на каждом слое). Обобщённый passthrough-механизм для диагностических
команд отклонён как преждевременная абстракция.

### src/vpn/mod.rs

- `pub async fn regen_client(&self, name: &str) -> Result<AddResult>` —
  валидация имени → `run(["regen", <name>])` → `existing_files(name)`.
  Семантика ошибок как у `add`: rc ≠ 0 → `ScriptFailed`; rc 0 без `.conf` →
  `Parse` (фоллбэк `existing_files`).
- `pub async fn regen_all(&self, reset_routes: bool) -> Result<bool>` —
  `run_capture(["regen"])`, при `reset_routes` добавляется `--reset-routes`.
  Возвращает `Ok(true)` при rc 0, `Ok(false)` при rc ≠ 0: скрипт ставит rc 1
  при частичных ошибках, когда часть клиентов уже перевыпущена — это
  «завершено с предупреждениями», а не отказ операции. Таймаут — увеличенный
  (`op_timeout_secs × 3`): массовый regen пропорционален числу клиентов.
  Новые ключи конфига не вводятся.
- `pub async fn diagnose(&self) -> Result<String>` — копия паттерна `check()`:
  `run_capture(["diagnose"])`, вывод возвращается независимо от кода выхода
  (ненулевой код у диагностики означает «найдены проблемы»).

### src/vpn/model.rs

- `pub fn format_expiry_badge(lang: Lang, now: i64, expiry: Option<i64>) ->
  Option<String>` — чистая функция рядом с существующей `format_expiry`.
  Возвращает компактную метку для кнопки списка: будущий срок — «⏳ 6д» /
  «⏳ 6d» (дни; при < 1 дня — часы «⏳ 5ч» / «⏳ 5h»), истёкший, но ещё не
  удалённый cron'ом — «⏳ истёк» / «⏳ expired», `None` — для бессрочных
  (`expiry == None`).

### Callbacks (src/bot/handlers.rs, parse_callback)

| Callback           | Действие                                            |
|--------------------|-----------------------------------------------------|
| `regen:<name>`     | перевыпустить одного клиента, прислать файлы        |
| `regen_all`        | показать экран подтверждения массового перевыпуска  |
| `regen_all_go`     | запустить `regen` (без сброса маршрутов)            |
| `regen_all_routes` | запустить `regen --reset-routes`                    |
| `diagnose`         | запустить диагностику, показать вывод               |

### src/bot/menu.rs

- `client_card`: добавить кнопку «🔄 Перевыпустить» (`regen:<name>`) в ряд к
  «📄 Конфиг» / «🗑 Удалить».
- `clients_list`: к тексту кнопки клиента добавляется метка из
  `format_expiry_badge` (данные — `client_expiry()` по каждому имени страницы;
  страница ≤ `per_page` элементов, чтение локальных файлов дёшево); внизу
  списка — кнопка «🔄 Перевыпустить всех» (`regen_all`).
- `main_menu`: кнопка «🩺 Проверка» становится рядом «🩺 Проверка | 🔬
  Диагностика» (`check` | `diagnose`).
- Новая клавиатура `confirm_regen_all(lang)`: «✅ Перевыпустить»
  (`regen_all_go`) / «🔀 Перевыпустить + сброс маршрутов» (`regen_all_routes`)
  / «⬅️ Отмена» (`menu`).

### src/i18n.rs

Новые строки (RU/EN, имена через `html_escape`):

- кнопки: `btn_regen`, `btn_regen_all`, `btn_diagnose`;
- подтверждение массового regen: текст объясняет, что файлы и QR всех клиентов
  будут перегенерированы с сохранением ключей, а вариант «+ сброс маршрутов»
  заменит индивидуальные `AllowedIPs` глобальным режимом сервера;
- прогресс: `regen_running` («⏳ Перевыпускаю…»), `regen_all_running`,
  `diagnose_running`;
- итоги: `regen_all_done` («✅ Все конфиги перевыпущены»),
  `regen_all_partial` («⚠️ Завершено, но с ошибками у части клиентов —
  проверьте логи сервера»), `diagnose_result` (обёртка `<pre>`, как
  `check_result`).

## UX-потоки

1. **Перевыпуск одного клиента.** Карточка → «🔄 Перевыпустить» →
   «⏳ Перевыпускаю…» → `send_client_files` (те же `.conf`/QR/URI, что у
   «📄 Конфиг») → «Готово» + главное меню. Подтверждение не требуется:
   операция неразрушающая (ключи и IP сохраняются).
2. **Массовый перевыпуск.** Список клиентов → «🔄 Перевыпустить всех» → экран
   подтверждения (3 кнопки) → «⏳ Перевыпускаю всех…» → итог
   (`regen_all_done` либо `regen_all_partial`). Файлы не рассылаются — только
   статус; свежий конфиг конкретного клиента забирается из его карточки.
3. **Диагностика.** Главное меню → «🔬 Диагностика» → «⏳ Диагностирую…» →
   вывод в `<pre>` с обрезкой до лимита Telegram-сообщения. Обрезка уже
   реализована инлайн в `Action::Check` (3500 байт с округлением до границы
   UTF-8-символа) — вынести её в общий хелпер и переиспользовать для
   `diagnose` (попутный целевой рефакторинг, поведение `check` не меняется).
4. **Метка срока.** `🟢 alice` → `🟢 alice ⏳ 6д` для временных; бессрочные —
   без изменений.

## Обработка ошибок

- `regen <name>`: клиент исчез (гонка) → rc 1 → `ScriptFailed` → стандартный
  `error_text`. rc 0 без `.conf` → `Parse` из `existing_files`.
- Массовый regen: rc ≠ 0 → `Ok(false)` → warning-сообщение; вывод скрипта в
  чат не отдаётся (секрет-гигиена), детали — в логах сервера.
- `diagnose`: код выхода игнорируется, вывод показывается всегда; пустой
  вывод → `Parse`.
- Метка ⏳: нечитаемый/отсутствующий expiry-файл → `client_expiry()` возвращает
  `None` → метки нет; ошибок для пользователя фича не генерирует.

## Тестирование

По слоям, как принято в проекте (стабы-скрипты для `Vpn`, чистые функции —
юнит-тестами):

- `vpn`: `regen_client` передаёт `regen <name>` и читает файлы; `regen_all`
  передаёт/не передаёт `--reset-routes` (по образцу `add_passes_psk_flag`);
  `regen_all` возвращает `Ok(false)` при rc 1, а не `Err`; `diagnose`
  возвращает вывод при rc 1.
- `model`: `format_expiry_badge` — дни, часы, истёкший, `None` для бессрочных,
  обе локали.
- `menu`: `client_card` содержит `regen:<name>`; `clients_list` содержит метку
  и `regen_all`; `confirm_regen_all` — три ожидаемых callback'а; `main_menu`
  содержит `diagnose`.
- `handlers`: `parse_callback` для всех новых префиксов.
- `i18n`: новые строки непусты в обеих локалях, имена экранируются.

## Порядок реализации

TDD, три независимых среза (каждый со своими атомарными коммитами):

1. Метка ⏳ в списке клиентов (`format_expiry_badge` → `clients_list`).
2. Диагностика (`Vpn::diagnose` → меню → handler).
3. Перевыпуск: сначала карточка (`regen_client`), затем массовый
   (`regen_all` + подтверждение).
