# AmneziaWG Telegram Bot — дизайн

**Дата:** 2026-07-10
**Статус:** утверждён к реализации

## 1. Цель

Telegram-бот на Rust для управления VPN-сервером [amneziawg-installer](https://github.com/bivlked/amneziawg-installer).
Позволяет доверенным администраторам добавлять/удалять клиентов и смотреть список
и статистику трафика прямо из Telegram, без SSH.

Бот — **тонкий слой** поверх штатного скрипта `manage_amneziawg.sh`: он не дублирует
логику установщика, а вызывает его команды с флагом `--json` и переводит результат
в удобный чат-интерфейс.

### Не входит в объём (YAGNI)
- Backup / restore через бота (делается штатным скриптом вручную).
- Управление несколькими серверами (бот живёт на том же VPS, что и VPN).
- Веб-панель, метрики, роли сложнее «admin / не-admin».
- Саморегистрация клиентов конечными пользователями.

## 2. Ключевые решения

| Решение | Выбор | Обоснование |
|---|---|---|
| Размещение | Тот же VPS, что и VPN | Прямой вызов скрипта, минимум движущихся частей |
| Взаимодействие с VPN | Вызов `manage_amneziawg.sh ... --json` | Переиспользование проверенной логики установщика |
| Библиотека | `teloxide` (+ `tokio`) | Де-факто стандарт: dispatcher, диалоги, inline-кнопки |
| Получение апдейтов | Long polling | Не нужен открытый порт, домен, TLS |
| Авторизация | Белый список `admin_ids` в конфиге | Просто, безопасно, без БД |
| UX | Inline-кнопки (меню) | Удобно с телефона, не нужно помнить синтаксис |
| Привилегии | По умолчанию root; hardened — юзер + sudoers | Простой старт, есть безопасный путь |

## 3. Архитектура

Один бинарник `awg-bot`, запускаемый через systemd. Асинхронный runtime `tokio`,
Telegram-слой на `teloxide` в режиме long polling. Все VPN-операции — через
`tokio::process::Command` (без шелла).

### Модули

```
src/
  main.rs         — старт: загрузка конфига, инициализация teloxide, запуск dispatcher
  config.rs       — чтение и валидация конфига (токен, admin_ids, пути, sudo_prefix, тайм-аут)
  auth.rs         — проверка user_id ∈ admin_ids (перед любым действием, включая callback)
  vpn/mod.rs      — фасад операций: add / remove / list / stats
  vpn/runner.rs   — запуск manage_amneziawg.sh: аргументы, тайм-аут, коды выхода, stdout/stderr
  vpn/model.rs    — структуры (Client, Stats, …) + serde-парсинг --json
  vpn/validate.rs — валидация имени клиента и срока действия (whitelist)
  bot/menu.rs     — построение inline-клавиатур
  bot/handlers.rs — команды, callback_query, диалоги (ввод имени/срока) с состоянием
  bot/render.rs   — форматирование ответов, отправка .conf / QR .png / vpn:// URI
  error.rs        — единый тип ошибки (thiserror)
```

### Принцип изоляции
- Слой `vpn/` ничего не знает о Telegram.
- Слой `bot/` ничего не знает о деталях скрипта.
- Граница между ними — типизированные структуры из `vpn/model.rs`.
- Бизнес-логика сосредоточена в `vpn/`, поэтому тестируется без Telegram API
  (runner подменяется на скрипт-заглушку).

### Поток данных — пример «Добавить клиента»

```
Telegram update → dispatcher → auth (проверка admin)
   → handler "add" → диалог: спросить имя → (опц.) спросить срок
   → vpn::add(name, expires) → validate → runner: <sudo> manage_amneziawg.sh add <name> [--expires=<v>] --json
   → парсинг JSON → пути к .conf / .png / .vpnuri в clients_dir
   → render: отправить документ + фото QR + текст с vpn:// URI
```

## 4. Возможности и UX

### Точка входа `/start`
Только для admin. Для остальных — «Доступ запрещён» + запись в лог.
Главное inline-меню:

```
🔐 AmneziaWG
├─ 👥 Клиенты        (список)
├─ ➕ Добавить
└─ 📊 Статистика
```

### ➕ Добавить (диалог с состоянием)
1. Бот: «Введите имя клиента» → ждёт текст.
2. Валидация имени (см. §6). При ошибке — понятное сообщение, повтор ввода.
3. Выбор срока кнопками:
   `Без срока · 1д · 7д · 14д · 30д · 90д · 180д · 365д · ✏️ Свой`
   - «✏️ Свой» → ввод вручную в формате `Nd` / `Nh` / `Nw` (напр. `10d`, `12h`, `3w`),
     валидируется и передаётся в `--expires`.
4. Вызов `add`, затем бот присылает **тремя сообщениями**: документ `<имя>.conf`,
   фото QR `.png`, текст с `vpn://…` URI.
5. Кнопка «⬅️ В меню».

### 👥 Клиенты
`list --json` → список; каждый клиент — кнопка. Тап открывает карточку:

```
client: alice
статус: активен · истекает: 2026-08-01
трафик: ↓ 1.2 GB  ↑ 340 MB
[ 📄 Конфиг ]  [ 🗑 Удалить ]
```

- «📄 Конфиг» — повторно прислать `.conf` / QR / URI.
- «🗑 Удалить» — подтверждение (`Точно удалить alice? Да / Отмена`), затем `remove`.
- Пагинация (◀️ ▶️), если клиентов много.

### 📊 Статистика
`stats --json` → сводка: число клиентов, суммарный трафик, кто онлайн
(по свежести handshake, если поле присутствует в выводе).

### Общие правила UX
- Долгая операция → «⏳ Выполняю…», затем правка сообщения результатом.
- Ошибка скрипта → дружелюбный текст, технические детали — только в лог.

## 5. Конфигурация

Файл `/etc/awg-bot/config.toml` (переопределяется `--config` / env `AWG_BOT_CONFIG`):

```toml
bot_token     = "123456:ABC..."        # или env AWG_BOT_TOKEN (приоритет у env)
admin_ids     = [11111111, 22222222]   # белый список Telegram user ID
manage_script = "/root/awg/manage_amneziawg.sh"
clients_dir   = "/root/awg"            # где лежат .conf / .png / .vpnuri
sudo_prefix   = "sudo"                 # "" если бот уже под root
op_timeout_secs = 60                   # тайм-аут одной операции
```

- Токен предпочтительно из env (в systemd — `EnvironmentFile` с правами `600`),
  чтобы не хранить секрет рядом с конфигом.
- При старте конфиг валидируется (fail-fast): непустой токен, непустой `admin_ids`,
  существующий и исполняемый `manage_script`.

## 6. Безопасность

- **Авторизация.** Каждый апдейт проходит `auth`: `user_id ∉ admin_ids` → отказ + лог.
  Проверяются и команды, и callback-и (не только `/start`).
- **Инъекции аргументов.** Имя клиента — whitelist-регэксп (латиница/цифры/`-`/`_`,
  ограниченная длина); срок — строгий формат `Nd|Nh|Nw`. Валидация **до** вызова.
  Аргументы передаются отдельными `Command::arg()` **без шелла** → shell-инъекция
  невозможна в принципе.
- **Привилегии.** По умолчанию сервис от root (проще, т.к. скрипт и `/root/awg/`
  требуют root). Hardened-вариант в README: отдельный юзер `awg-bot` + sudoers
  `awg-bot ALL=(root) NOPASSWD: /root/awg/manage_amneziawg.sh`, `sudo_prefix="sudo"`.
- **Секреты.** `.conf` и QR содержат приватные ключи → шлём только админу в личку,
  содержимое не логируем.

## 7. Обработка ошибок

Единый тип `Error` (`thiserror`). Принцип: **ошибка операции никогда не роняет бота** —
хэндлеры возвращают `Result`, ошибки ловятся сверху, пользователь получает
дружелюбный текст, бот продолжает работу.

| Ошибка | Причина | Реакция |
|---|---|---|
| `Config` | нет токена / пустой admin_ids / нет скрипта | fail-fast при старте с понятным сообщением |
| `ScriptFailed { code, stderr }` | ненулевой код выхода | «❌ Операция не удалась»; stderr в лог |
| `Timeout` | дольше `op_timeout_secs` | процесс убивается; «⏳ Превышено время ожидания» |
| `Parse` | неожиданный `--json` | «Не удалось разобрать ответ сервера»; сырой вывод в лог |
| `Telegram` | сетевые сбои polling | ретраи teloxide; ошибки отправки логируем, бот жив |

**Логирование:** `tracing` + `tracing-subscriber` → journald. Пишем: старт/стоп,
попытки доступа (кто, разрешён/отклонён), запущенные операции (без секретов),
ошибки со stderr скрипта.

## 8. Тестирование

- **Юнит-тесты:** валидация имени/срока (включая попытки инъекций); парсинг `--json`
  в структуры на зафиксированных примерах вывода; логика `auth`.
- **Тест runner-а:** подмена `manage_script` на скрипт-заглушку (echo фиксированного
  JSON / заданный код выхода / задержка) → проверка разбора успеха / ошибки / тайм-аута
  без реального VPN.
- **Ручной smoke-тест:** на тестовом сервере — `/start`, add → получить конфиг, list,
  remove. Чек-лист в README.

## 9. Деплой

- Статический бинарник (по возможности `x86_64-unknown-linux-musl`) →
  `/usr/local/bin/awg-bot`.
- systemd-юнит `awg-bot.service`: `Restart=on-failure`,
  `EnvironmentFile=/etc/awg-bot/env`, hardening (`NoNewPrivileges` и т.п. — с учётом
  доступа к `clients_dir`).
- README: сборка, установка, настройка sudoers (hardened), запуск, smoke-чек-лист.
- Опционально (за рамками MVP): `install.sh` для установки одной командой.

## 10. Открытые вопросы для реализации
- Точная схема `--json` для `list` / `stats` / `add` — зафиксировать по фактическому
  выводу скрипта на этапе реализации (структуры в `vpn/model.rs` подгоняются под неё).
- Наличие поля свежести handshake в выводе `stats` — определяет раздел «кто онлайн».
